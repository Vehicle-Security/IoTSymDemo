# CO3 原理速读

论文：**CO3: Concolic Co-execution for Firmware**  
会议：USENIX Security 2024  
链接：https://www.usenix.org/conference/usenixsecurity24/presentation/liu-changming

这份笔记的目标不是复述论文，而是帮助没读过论文的人快速理解 CO3 的基本思路，以及它和本仓库 `src-demo/` 的关系。

## 一句话理解

CO3 想解决的问题是：**怎样对 MCU 固件做符号/混合执行分析，同时避免硬件仿真不准、调试接口太慢的问题？**

它的答案是：

```text
真实 MCU 负责跑固件，保证硬件行为真实；
工作站负责算符号约束，保证计算资源充足；
两者通过普通串口/USB 只交换少量必要信息。
```

论文把这种方式叫 **Concolic Co-execution**，可以翻译成“混合执行式协同执行”。

## 先理解三个词

### 1. Concrete Execution：具体执行

给程序一个真实输入，然后程序按这一个输入跑一遍。

比如输入：

```text
buf[0] = 0xA5
```

程序判断：

```c
if (buf[0] == 0xA5)
```

结果就是 `true`。这就是具体执行。

优点是快、真实。缺点是一次只能看到一条路径。

### 2. Symbolic Execution：符号执行

不把输入看成固定值，而是看成变量。

比如：

```text
buf[0] = x
```

程序判断：

```c
if (x == 0xA5)
```

符号执行会记录一个约束：

```text
x == 0xA5
```

然后交给求解器，比如 Z3，求出能走这条路径的输入。

优点是能主动推导新输入。缺点是状态爆炸，而且在 MCU 上很难准确模拟外设。

### 3. Concolic Execution：混合执行

Concolic = concrete + symbolic。

程序先拿一个真实输入跑起来，同时记录这条路径上的符号约束。跑完后，把某个分支条件反过来求解，生成下一个输入。

例子：

```text
这次走了：buf[0] == 0xA5
下次想试：buf[0] != 0xA5
求解器给出：buf[0] = 0x00
```

这就能从一个输入不断扩展到更多路径。

## 固件分析为什么难

普通桌面程序依赖操作系统接口，比如文件、socket、系统调用。虽然也复杂，但至少有通用抽象。

MCU 固件不一样，它常常直接和硬件打交道：

- **MMIO**：读写某个内存地址，其实是在读写外设寄存器。
- **Interrupt**：外设在某个时间触发中断，固件进入 ISR。
- **DMA**：外设或 DMA 控制器直接把数据搬到内存，CPU 不一定参与。

这导致两条传统路线都有问题。

## 传统路线的问题

### 路线 A：全仿真

把固件搬到 QEMU、Unicorn 之类环境里跑，然后模拟外设。

问题是外设模型很难做准。模型不准会导致：

- 程序状态不真实。
- 触发假的 bug。
- 漏掉真实 bug。
- 固件甚至启动不了。

### 路线 B：Hardware-in-the-Loop

让真实 MCU 参与，工作站通过调试接口控制它。

典型过程可能是：

```text
暂停 MCU -> 设置断点 -> 继续运行 -> 等命中断点 -> 读取状态
```

这种方法真实，但同步成本很高。论文指出，很多 HiL 方案被调试接口拖慢，还可能依赖昂贵或定制化硬件。

## CO3 的核心转向

CO3 不再让工作站“控制” MCU，而是让 MCU 主动“报告”必要信息。

可以类比成：

```text
传统 HiL：工作站像调试器，频繁打断 MCU 问：你现在怎样？
CO3：MCU 像合作程序，跑到关键点时主动说：这个值你需要知道。
```

这样有三个好处：

1. 固件真的跑在 MCU 上，外设行为是真实的。
2. 不需要模拟所有硬件事件。
3. 不需要通过慢速调试协议频繁暂停和恢复 MCU。

## CO3 的整体流程

CO3 分成离线准备和在线执行两阶段。

### 离线准备

输入是固件源码。

CO3 的编译器做两件事：

```text
compile(source)
  -> 带插桩的 MCU 固件二进制
  -> 工作站使用的 SVFG
```

其中：

- 插桩二进制烧到 MCU 上运行。
- SVFG 放在工作站上，用来恢复符号公式。

### 在线执行

在线阶段是一个循环：

```text
1. 工作站给 MCU 一个 seed input
2. MCU 用真实硬件运行插桩固件
3. MCU 通过串口/USB 报告必要运行信息
4. 工作站根据 SVFG + 运行信息构造路径约束
5. Z3 求解新输入
6. 新输入再发给 MCU
```

这就是“真实执行 + 符号推理”的协同。

## SVFG 是什么

SVFG 全称是 **Symbolic Value Flow Graph**，可以理解为“符号值流图”。

初学者可以把它想成：

```text
工作站提前拿到的一张公式骨架图。
```

比如某个分支条件其实来自：

```text
input_byte + ram_value < mmio_value
```

编译时能提前知道：

- 有一个加法。
- 有一个小于比较。
- 比较结果会进入分支。
- 某些常量已经确定。

但编译时不知道：

- 运行时 RAM 地址是多少。
- 某次 MMIO 读到了什么具体值。
- 分支这次到底走 true 还是 false。

所以 CO3 的思路是：

```text
公式结构提前放在工作站；
运行时缺的值由 MCU 报告；
工作站把两者拼起来。
```

这样 MCU 不需要发送每条指令，也不需要自己构造复杂符号表达式。

## MCU 上到底记录什么

CO3 插桩后，MCU 主要做轻量级工作：

1. 标记哪些内存字节来自输入，是“符号”的。
2. 在符号数据被读写时，必要时报告地址。
3. 当符号值和具体值一起计算时，报告那个具体值。
4. 分支发生时，报告实际走了 true 还是 false。
5. 输入通道开始时，报告输入被放到了哪里。

注意：MCU 不负责跑 Z3，也不负责保存完整符号表达式。

## Shadow Memory：影子内存

符号执行需要知道“这个值是不是由输入影响来的”。

传统做法是影子内存：每个真实内存字节旁边，都有一个影子状态。

CO3 有两份影子内存：

| 位置 | 存什么 |
|---|---|
| MCU | 只存每个字节是否符号化，通常 1 bit 就够 |
| 工作站 | 存真正的符号表达式 |

这样设计是为了省通信。

如果 MCU 每次读写 RAM 都报告，串口会被打爆。CO3 让 MCU 先用本地影子内存判断：

```text
这次读写完全不涉及符号数据 -> 不报告
这次读写会影响符号状态 -> 报告
```

这就是论文里 CO3 能比传统 HiL 快很多的关键原因之一。

## 外设输入怎么符号化

MCU 的输入不一定来自 `stdin`，而是来自真实外设。论文讨论了几类常见入口。

### MMIO

外设数据常常通过某个 data register 进入固件。

CO3 的 monitor 会把工作站给的输入喂给这个寄存器，同时让工作站知道对应地址，这样两边都能把它看成符号输入。

### DMA

DMA 会把外设数据搬到某个目标 buffer。

CO3 会定位这个目标 buffer，把 buffer 内容符号化，并把地址告诉工作站。

### Interrupt

中断仍然由真实 MCU 触发。CO3 只需要给 ISR 也插桩，让 ISR 像普通函数一样报告符号相关信息。

为了避免运行时状态被中断打断，CO3 在进入自己的 runtime 时会临时关中断，退出后再打开。

### Designated Buffer

论文还支持一种更实用的方式：直接指定某个 buffer 为输入。

很多固件都会先从 UART、DMA、USB 等地方收数据，然后放进一个 buffer，再做协议解析。对测试来说，真正重要的是后面的解析逻辑，所以直接指定这个 buffer 往往更简单。

本仓库的 `src-demo/` 就更接近这种教学模式：把 `input.bin` 读进 `buf`，然后分析 `process_packet(ctx, buf, len)`。

## 工作站做什么

工作站负责重活：

1. 接收 MCU 的运行消息。
2. 用 SVFG 找到对应的公式骨架。
3. 填入运行时具体值。
4. 构造当前路径约束。
5. 选择某个分支条件取反。
6. 调用 Z3 生成新输入。

一句话：**MCU 报事实，工作站建公式。**

## 为什么 CO3 快

论文里的性能提升主要来自几个设计选择：

1. **不转发所有硬件事件**  
   MMIO、中断、DMA 都在真实 MCU 上自然发生，不需要工作站模拟。

2. **不使用调试接口同步每一步**  
   MCU 主动发消息，工作站可以边收边处理。

3. **不发送完整执行轨迹**  
   SVFG 已经在编译时提供公式骨架，运行时只发缺失部分。

4. **MCU 只做符号状态传播**  
   对 MCU 来说，很多符号计算可以简化成“这个结果是否受输入影响”的布尔传播。

5. **本地影子内存减少无意义通信**  
   只有符号相关的内存读写才需要报告。

论文评估中，CO3 相比传统 HiL 符号执行方案快了约三个数量级，并且和工作站上的 SymCC 相比也有竞争力。

## SHACO：CO3 + Fuzzing

CO3 本身是混合执行器，擅长“翻转分支，突破难条件”。

但找 bug 时，fuzzing 也很重要：

- fuzzing 很快，适合扫大量容易到达的状态。
- concolic execution 较慢，但适合解开很窄的条件。

论文把 CO3 和 SHiFT 结合，形成 **SHACO**：

```text
SHiFT：负责快速 fuzzing、覆盖率反馈、崩溃检测
CO3：负责对有价值输入做分支翻转，生成更聪明的新输入
```

实验中，SHACO 在多个固件样本上更快找到已知 bug，并发现了新的固件 bug。

## CO3 的边界

CO3 很强，但不是万能的。

### 需要源码

CO3 依赖 LLVM 级插桩，所以需要固件源码。它不是黑盒二进制分析方案。

### 需要能烧录插桩固件

目标场景更像固件开发者、安全团队、设备厂商在测试自己的固件，而不是只拿到一个未知设备黑盒。

### 会引入运行时和内存开销

CO3 插桩后会增加 Flash、RAM 和运行时间开销。论文给了不同模式：

| 模式 | 思路 | 特点 |
|---|---|---|
| Full-On | 启用 MCU 侧影子内存和通信优化 | 最快，但 RAM 开销更大 |
| Shadowless | 不使用 MCU 侧 RAM 影子内存 | RAM 省很多，但通信和时间开销更高 |
| Report-All | 尽量全部报告 | 简单但最慢，不推荐 |

## 和本仓库 demo 的关系

本仓库不是 CO3 的完整复现，而是一个**教学用最小原型**。

对应关系如下：

| CO3 论文 | 本仓库 `src-demo/` |
|---|---|
| 真实 MCU 跑插桩固件 | 本机运行 `miniiot_instrumented` |
| 串口/USB 发送运行消息 | 写入 `traces/*.jsonl` |
| LLVM 自动插桩 | 手工/LLM 插桩 |
| SVFG 公式骨架 | README 和 trace 里的简化条件说明 |
| 工作站符号引擎 | `analysis.py` 的简化 trace 分析 |
| Z3 求解新输入 | 当前只预留简单 Z3 检查，不完整生成新输入 |
| MMIO/DMA/ISR 输入 | `input.bin` 指定协议输入 buffer |

所以这个 demo 的价值是先讲清楚 CO3 的核心感觉：

```text
原始固件逻辑
  -> 加插桩
  -> 运行输入
  -> 记录分支、比较、内存检查、bug
  -> 用 trace 理解路径条件
```

等这个流程理解后，再看论文里的 LLVM、SVFG、影子内存、串口协议和真实 MCU，就不会一下子被系统复杂度淹没。

## 用一个 MiniIoT 例子串起来

在 `src-demo/miniiot_instrumented.c` 中，原始逻辑会检查：

```c
buf[0] == 0xA5
buf[1] == 0x5A
buf[2] == 0x01
checksum 是否正确
msg_type 是哪一种
```

插桩后，每个关键判断都会记录：

```text
比较了什么
实际值是多少
分支走 true 还是 false
最后返回什么
```

当输入是 `009_write_config_oob` 时，trace 会告诉我们：

```text
进入 WRITE_CONFIG
准备写 config[index]
index = 16
config 长度 = 16
下标不合法
记录 OOB_CONFIG_WRITE
返回 -100
```

这就是 CO3 思路的迷你版：**让程序正常跑，但把路径上的关键事实留下来。**

## 最重要的直觉

读 CO3 时抓住这几个直觉就够了：

1. **不要模拟 MCU，直接用真实 MCU。**
2. **不要频繁调试暂停，让固件主动报告。**
3. **不要报告所有东西，只报告符号推理缺的东西。**
4. **复杂公式放工作站，MCU 只做轻量标记。**
5. **concolic execution 不是替代 fuzzing，而是帮 fuzzing 突破难分支。**

这也是本仓库 demo 想表达的最小闭环。

我们为了简化设计，模拟实现了一个小的IoT固件，手工插桩记录了输入、分支条件和漏洞事件。并将Trace保存为文件，来模拟MCU和PC之间传符号信息。
